接口返回的回答里，能够带上来源片段或来源信息。

适用范围
当前项目技术栈：Node.js + Express
当前数据库阶段：已有已入库的真实文档
当前检索阶段：已能在脚本中手动 query
当前目标：把“检索结果”真正接进“问答接口”
今天做完后，你应该达到的结果
能说清楚 retrieval API 在 RAG 里的位置
能根据用户问题检索出 top-k chunks
能把检索结果拼进 prompt
能让模型基于这些片段回答
能把引用信息一起返回给前端
先用一句话理解
今天要做的，是把“数据库会检索”升级成“接口会先检索，再基于检索结果回答”。

什么叫 retrieval API
你今天先把 retrieval API 理解成这条链路：

用户提问
后端把问题拿去向量检索
取回最相关的 top-k chunks
把这些 chunk 作为上下文拼进 prompt
调模型生成答案
返回答案和引用信息
一句话版：

“不是模型凭空回答，而是先拿到相关片段，再基于片段作答。”

今天为什么一定要把检索和回答接起来
因为 Day 17 和 Day 18 只完成了：

数据已入库
数据可检索
但真正的 RAG 还差最关键的一步：

检索结果如何进入模型上下文
如果你不做这一步，那你的系统还只是：

一个向量库
一个普通聊天接口
还不是一个真正的最小 RAG 闭环。

Day 19 的意义，就是把“检索”和“生成”第一次真正接起来。

先建立今天的最小心智模型
今天你最适合把后端逻辑理解成下面这段伪流程：

接收用户问题
先对问题做 query
拿到 top-k 检索结果
把结果整理成 context blocks
构造 prompt
调模型回答
把答案和引用一起返回
最关键的原则是：

先检索，再回答
引用不要丢
prompt 里只放必要上下文
推荐学习和实战顺序
模块 1：先定义 retrieval API 的输入输出（40min）
这一阶段你先把接口想清楚。

输入最少应包含：

question
可选 topK
输出最少应包含：

answer
citations
可选 retrievedChunks
这一阶段最关键的是：

“你要先清楚接口到底返回给前端什么，而不是先急着写 prompt。”

模块 2：实现纯检索部分（1h）
先不要急着接模型。

你应该先做一条只负责 query 的逻辑：

接收 question
调用 collection 的 query
返回最相关的若干 chunks
先确认：

top-k 参数有效
返回结果格式清楚
引用字段能从 metadata 中拿出来
如果这一步不稳，后面的生成也没有基础。

模块 3：把检索结果整理成 prompt context（1.5h）
当检索结果已经拿到后，再做 prompt 组装。

建议你把检索结果整理成这种结构：

片段编号
来源信息
正文内容
这样后面模型回答时，更容易引用和对齐。

这一阶段你最该学会的是：

检索结果不能一股脑原样塞进去
context 需要结构化
prompt 需要明确要求“基于以下片段回答”
模块 4：调用模型生成答案，并返回引用（1.5h）
这一阶段把真正的 RAG 闭环接起来。

你需要做的事：

把用户问题和检索上下文一起交给模型
让模型生成回答
同时把引用信息一起返回前端
今天最稳妥的做法不是让模型自由发挥，而是：

返回结构化 citations
前端直接展示 source/page/chunkId
不把“是否保留引用”交给模型自己决定
模块 5：做一轮完整问答验证（40min）
你至少要测试这几类问题：

文档中明确写过的
需要检索多个 chunk 才能回答的
文档中没有答案的
这一阶段的重点是：

回答是否基于检索内容
引用是否能对应到真实 chunk
没答案时模型有没有胡编
今天最值得注意的常见坑
坑 1：把 query 结果直接原样塞给模型
检索结果需要整理，否则 prompt 会很乱，模型也更容易引用错位。

坑 2：回答里有引用，但接口本身没返回 citations
你不应该把“引用展示”完全交给自然语言。更稳的做法是接口单独返回结构化引用字段。

坑 3：top-k 固定不变且不记录
今天开始你就应该把 top-k 当成一个实验参数，而不是写死后不管。

坑 4：只测有答案的问题
真正的 RAG 很容易在“无答案”场景里暴露问题，所以今天至少要测一两个检索不到的题。

今天的验收标准
已有 retrieval API
API 能先做 top-k 检索
检索结果能进入 prompt
模型能基于检索片段回答
返回结果里保留了引用信息
至少完成一轮真实问答验证
建议的测试问题
为了提高成功率，今天优先用：

文档里明确定义过的问题
文档里有直接说明或结论的问题
可以明显验证出处的问题
例如：

“三相不平衡常见原因是什么？”
“路由守卫通常在什么时候使用？”
“这份文档里有没有提到无功补偿？”
“文档里有没有提到 React Server Components？”
这些问题适合你验证：

检索能不能命中
引用能不能落到来源
无答案场景是否会胡说

一份 Day 19 笔记
一个 retrieval API
一版 prompt 模板
一组带引用的问答结果
一次“有答案 / 无答案”对比记录
最后给你的实操建议
今天最重要的不是把回答写得多漂亮，而是保证“先检索，再回答，并且引用不丢”。
