# Day 24｜2026-04-18｜6h

## 今日主题

- 完成后端上传接口
- 串联上传与入库流程
- 加失败重试或错误提示

验收：

上传文档后能看到它进入知识库。

## 适用范围

- 当前项目技术栈：`Node.js + Express + Vue`
- 当前前端状态：上传页 UI 已经成型
- 当前后端能力：已有文档解析、切片、向量库入库基础
- 当前目标：把“页面上传文件”真正接到“后端入库知识库”的主链路上

## 今天做完后，你应该达到的结果

- 能提供一个正式的上传接口
- 能接住前端上传的 PDF/TXT 文件
- 能串起解析、切片、embedding、入库流程
- 能把处理状态回给前端
- 能在失败时给出明确错误或重试路径

## 先用一句话理解

今天要做的，是把“上传页看起来能用”升级成“用户上传一份文档后，它真的会进入知识库”。

## 什么叫“上传与入库串起来”

今天最核心的不是单独完成上传，也不是单独完成入库，而是让这条链路完整跑通：

1. 前端上传文件
2. 服务端保存临时文件或读取文件流
3. 服务端解析文本
4. 服务端切片
5. 服务端写入向量库
6. 服务端更新文档状态
7. 前端能看到最终结果

一句话版：

“不是文件到了服务器就算成功，而是它最终进入知识库才算成功。”

## 今天为什么这一步最关键

因为从 Day 24 开始，你的产品第一次开始具备真正的业务入口。

如果这一步不稳，后面所有问答体验都会失去基础：

- 问答页没有可问的文档
- 文档列表页没有真实数据
- 引用展示没有来源
- 多轮问答没有知识上下文

Day 24 的意义，就是把“前端上传”接成“知识库新增内容”。

## 先建立今天的最小后端链路

你今天最适合把上传接口理解成两层：

### 第一层：HTTP 接口层

- 接收 multipart/form-data
- 校验文件
- 返回文档 ID 和当前状态

### 第二层：文档处理层

- 解析文本
- 切 chunk
- 写入向量库
- 更新状态

最稳的思路是把这两层分开，不要把所有代码都堆在一个路由函数里。

## 推荐学习和实战顺序

### 模块 1：先定义接口输入输出（40min）

今天最先要做的是接口契约，而不是立刻接 `multer`。

最推荐的最小返回结构：

```json
{
  "ok": true,
  "data": {
    "documentId": "doc_001",
    "fileName": "demo.pdf",
    "status": "processing"
  }
}
```

失败时建议保持统一：

```json
{
  "ok": false,
  "error": "Only PDF/TXT files are supported"
}
```

### 模块 2：实现上传入口和文件校验（1.5h）

这一阶段重点是把文件可靠地接住。

你至少应该处理：

- 空文件
- 不支持的类型
- 文件过大
- 服务端保存失败

这里最适合统一校验规则：

- 只允许 `.pdf` `.txt`
- 统一大小限制
- 统一错误信息格式

### 模块 3：串联入库流程（2h）

当文件接住后，再把 Day 16 到 Day 18 的文档链路接进来。

主流程建议拆成：

- `uploadDocument`
- `parseDocument`
- `chunkDocument`
- `ingestChunks`
- `updateDocumentStatus`

你今天最该避免的是：

- 把解析、切片、入库、状态更新全部堆在路由里

更稳的方式是：

- 路由只负责接请求和返回结果
- service 负责真正的文档处理

### 模块 4：补状态查询与错误提示（1h）

如果上传接口只返回“成功”，用户还是不知道文档到底入没入库。

所以今天最值得补的是状态接口或状态字段：

- queued
- processing
- ready
- failed

前端至少要能轮询或刷新看到文档状态变化。

### 模块 5：补失败重试路径（40min）

今天不一定非要做复杂任务队列，但至少要做到：

- 失败时给出明确提示
- 前端可以重新上传
- 后端日志能定位失败在哪一步

最稳的 Day 24 目标不是“复杂后台任务系统”，而是“失败时不会悄悄无声”。

## 今天最值得注意的常见坑

### 坑 1：上传成功就返回 ready

这会误导前端。真正 ready 应该是入库完成之后。

### 坑 2：接口成功和业务成功混淆

HTTP 200 只说明请求收到了，不说明向量库写入成功。

### 坑 3：失败信息不具体

如果只返回“upload failed”，后面很难排查到底是上传、解析还是入库失败。

### 坑 4：前端页面和后端状态名字不统一

今天最好把状态名统一成一套，不然后续会一直对不上。

## 今天的验收标准

- 前端能上传 PDF/TXT
- 后端能接住文件并开始处理
- 文档能经过解析、切片、入库流程
- 前端能看到文档进入知识库或明确失败
- 出错时至少有一种可继续操作的路径

## 建议的测试场景

- 上传一个合法 TXT 并成功入库
- 上传一个非法类型文件
- 上传一个损坏文件导致解析失败
- 重复上传同一文件
