# Day 23｜2026-04-17｜6h

## 今日主题

- 完成上传页 UI
- 支持拖拽上传、文件校验、进度状态
- 显示文档处理状态

验收：

用户能上传 PDF/TXT。

## 适用范围

- 当前项目技术栈：`Node.js + Express + Vue`
- 当前页面规划：已有上传页、问答页、文档列表页的产品骨架
- 当前后端状态：上传接口还未完全接好，但页面交互可以先搭起来
- 当前目标：先把上传页做成一个清晰、可演示、可继续接后端的正式页面

## 今天做完后，你应该达到的结果

- 能完成一个正式的上传页而不是临时调试页面
- 能支持点击选择和拖拽上传两种入口
- 能在前端先做文件类型和大小校验
- 能展示上传中、处理中、成功、失败等状态
- 能让 Day 24 直接把后端接口接上，不需要再大改 UI

## 先用一句话理解

今天要做的，不是把上传功能彻底打通，而是先把“用户上传知识库文档时应该看到什么、怎么操作、状态怎么反馈”完整做出来。

## 什么叫“上传页可用”

一个真正可用的上传页，至少要让用户明确知道：

- 我可以上传什么文件
- 我正在上传什么文件
- 当前进度到哪一步了
- 上传失败是为什么
- 文档最终有没有进入处理流程

一句话版：

“上传页不是一个按钮，而是一整套让用户理解当前状态的交互面板。”

## 今天为什么要先做上传页 UI

因为 Day 24 会接后端上传接口，如果今天 UI 不先定清楚，明天很容易出现：

- 请求体不知道怎么组织
- 成功/失败状态没有落点
- 文档处理状态无处显示
- 前后端联调时页面频繁返工

Day 23 的意义，就是把上传这件事从“能选文件”升级成“像产品一样可交互”。

## 先建立今天的最小页面心智模型

上传页最少包含 4 个区域：

1. 上传说明区
2. 拖拽/选择文件区
3. 当前文件状态区
4. 处理结果区

你最推荐先支持的状态有：

- idle：未选择文件
- validating：正在校验
- uploading：正在上传
- processing：后端正在入库
- success：已进入知识库
- error：失败

一句话版：

“今天不是做一个输入控件，而是做一张有状态流转的上传页面。”

## 推荐学习和实战顺序

### 模块 1：先定页面状态，不急着写样式（50min）

今天最先要确定的不是颜色，而是状态机。

你应该先定清楚：

- 页面默认显示什么
- 选中文件后显示什么
- 校验失败时显示什么
- 上传中显示什么
- 后端处理中显示什么
- 成功后显示什么

这一阶段你最适合先写一个前端状态枚举，例如：

- `idle`
- `invalid`
- `ready`
- `uploading`
- `processing`
- `success`
- `error`

### 模块 2：完成拖拽和选择文件入口（1.5h）

上传页最关键的交互不是按钮，而是入口清晰。

你今天应该实现：

- 点击上传区域触发文件选择
- 支持拖拽文件进入区域
- 拖拽悬停时有高亮反馈
- 拖拽离开时恢复默认状态

今天最稳的策略是：

- 先只支持单文件上传
- 先只接受 PDF/TXT
- 先不做复杂队列管理

### 模块 3：做文件校验和状态展示（1.5h）

这一阶段重点不是请求，而是“用户为什么能继续下一步”。

你应该先做这些校验：

- 文件类型是否为 `.pdf` 或 `.txt`
- 文件大小是否超过你设定的上限
- 是否重复选择同一个文件

校验通过时，页面至少展示：

- 文件名
- 文件大小
- 文件类型
- 可点击开始上传或自动进入上传

校验失败时，页面至少展示：

- 明确错误原因
- 如何修复

### 模块 4：先用 mock 状态打通完整交互（1.5h）

今天后端接口还没完全接好，所以最稳的方式是：

- 先用前端 mock 或定时器模拟状态变化

例如：

- 点击上传后 1 秒进入 `uploading`
- 再过 1 秒进入 `processing`
- 再过 1 秒进入 `success`

如果你今天把这个链路做好，Day 24 只需要把 mock 换成真实接口。

### 模块 5：补可读反馈和页面细节（1h）

今天上传页最值得补的细节有：

- 支持重新选择文件
- 正在上传时禁用重复提交
- 成功后展示“去文档列表”或“去问答页”
- 失败后展示“重试”

这一阶段的原则是：

- 先让状态清楚
- 再考虑视觉精致

## 今天最值得注意的常见坑

### 坑 1：只做上传框，不做状态区

这样用户只能点，却不知道系统到底在干什么。

### 坑 2：拖拽上传和点击上传逻辑分成两套

更稳的方式是让它们最后都走同一套文件处理函数。

### 坑 3：文件校验只在后端做

前端也必须先做一次基础校验，否则体验很差。

### 坑 4：处理中和上传中混为一谈

上传完成不等于已经入库完成，这两个状态最好分开显示。

## 今天的验收标准

- 页面能让用户选择 PDF/TXT
- 页面支持拖拽上传
- 文件校验失败时能有明确提示
- 页面能展示上传中和处理中状态
- 页面整体已经像正式产品页面，而不是测试控件
