# Day 27｜2026-04-21｜5h

## 今日主题

- 做多轮对话
- 控制历史长度，避免上下文污染
- 测试追问是否还能正确检索

验收：

支持连续追问 3~5 轮。

## 适用范围

- 当前项目技术栈：`Node.js + Express + Vue`
- 当前问答状态：单轮提问已经可用，回答支持流式和引用
- 当前后端能力：已有 retrieval / answer 主链路
- 当前目标：让用户可以围绕同一份文档连续提问，而不是每次都从零开始

## 今天做完后，你应该达到的结果

- 能支持 3 到 5 轮连续对话
- 能在前端保留对话历史
- 能在后端合理组织历史上下文
- 能避免历史越来越长导致污染回答
- 能验证追问场景下检索是否仍然正确

## 先用一句话理解

今天要做的，是把“单次问答”升级成“用户能围绕同一主题继续追问”的连续对话体验。

## 什么叫“多轮对话可用”

不是简单地把旧消息全部带回去就算多轮。

真正可用的多轮对话至少要解决：

- 前端如何展示上下文
- 后端如何传历史消息
- 历史保留多少轮
- 追问是否仍然围绕当前知识库
- 检索是否会被无关历史干扰

一句话版：

“多轮的难点不在于把历史留着，而在于保留有用历史、控制无用历史。”

## 今天为什么一定要控制历史长度

如果你不控制历史，最容易发生的事就是：

- 历史越来越长，提示词膨胀
- 早期问题污染后续问题
- 模型开始参考旧答案而不是新检索结果
- 追问越多，结果越飘

Day 27 的意义，就是让连续追问可用，而不是让上下文无限堆积。

## 先建立今天的最小多轮链路

你今天最适合把多轮对话理解成这条链路：

1. 前端保存消息列表
2. 用户继续追问
3. 前端带上有限历史发送请求
4. 后端根据当前问题继续检索
5. 模型在“有限历史 + 新检索结果”下作答
6. 前端追加新一轮问答

一句话版：

“多轮对话不是只靠历史，而是历史和新检索一起工作。”

## 推荐学习和实战顺序

### 模块 1：先完成前端多轮消息保留（40min）

今天先保证前端消息列表能持续追加，而不是每轮刷新覆盖。

你最少要支持：

- 保留所有 user / assistant 消息
- 正确显示每轮引用
- 每次发送都在当前列表尾部追加

### 模块 2：设计后端历史输入策略（1.5h）

今天最关键的不是“全量带历史”，而是“有限带历史”。

最稳的 Day 27 策略是：

- 保留最近 3 到 5 轮
- system prompt 固定
- 每轮依然重新做检索

你今天不一定要做复杂 memory，总之先保证：

- 历史不会无限增长
- 当前问题仍然以当前检索结果为主

### 模块 3：控制上下文污染（1.5h）

这一阶段最值得你关注的是：

- 模型会不会只顺着上一轮答案说
- 会不会忽略新检索
- 会不会因为历史太长而跑偏

最稳的做法通常是：

- 历史只保留最近几轮对话
- prompt 明确要求优先依据当前检索结果回答

### 模块 4：重点测试追问场景（1h）

Day 27 的测试不应只测独立问题，而应该测连续追问，例如：

1. “这份文档主要讲什么？”
2. “你刚才提到的第二点展开说一下。”
3. “它对应的原文在哪一页？”
4. “那这份文档有没有提到相关限制？”

这类题能测试：

- 指代是否还能接上
- 检索是否还能命中
- 引用是否还能对应

### 模块 5：补清空会话和重开对话能力（40min）

多轮对话做出来以后，最好补两个能力：

- 清空当前会话
- 从新会话重新开始

这样用户就不会被旧历史长期绑定。

## 今天最值得注意的常见坑

### 坑 1：把所有历史全部发给后端

短期可用，长期一定会乱。

### 坑 2：追问时不重新检索

这样会越来越像普通聊天，而不是 RAG 问答。

### 坑 3：引用不跟随每轮消息保存

后面多轮时最容易错位。

### 坑 4：只测第一轮，不测第三轮第四轮

多轮问题恰恰会在后几轮暴露出来。

## 今天的验收标准

- 前端支持连续问答 3 到 5 轮
- 后端能带有限历史继续回答
- 每轮依然会做当前问题的检索
- 追问不会明显污染检索和回答
- 多轮体验已经达到可演示水平

## 建议的测试问题

- 先问总述，再追问细节
- 先问结论，再追问出处
- 先问是否提到某概念，再追问具体片段

今天最重要的是测试“连续性”，不是单轮准确率。
