先用一句话理解
今天要做的，是把“手工造几条文本”升级成“真实文档经过解析、切片、embedding、入库”的完整流程。

什么叫 ingestion 流程
你今天先把 ingestion 理解成一条固定流水线：

用户上传文件
服务端读取文件内容
解析出纯文本
按规则切成多个 chunk
为每个 chunk 准备 metadata
生成 embedding
写入向量库
一句话版：

“ingestion 就是把原始文档处理成可检索 chunk，并把它们规范地写进知识库。”

今天为什么一定要单独做 ingestion
因为真正的 RAG 系统不是手工往数据库里写 10 条文本，而是靠这条流水线接收真实文档。

如果 ingestion 不稳定，后面所有问题都会出现：

文档内容丢失
chunk 切分混乱
metadata 不完整
检索结果无法回溯来源
引用无法落到具体页面或片段
Day 18 的意义，就是把“知识库的数据入口”真正建立起来。

先建立今天的最小心智模型
你今天最适合把 ingestion 理解成两个阶段：

阶段 1：文档理解前处理
这一段负责：

上传
读取
解析
切片
阶段 2：向量化和入库
这一段负责：

embedding
写入 collection
保存 metadata
你今天不一定要支持很多文件格式，但至少要把其中一种做稳。

今天最推荐的输入文档策略
为了提高成功率，今天不要一开始就接复杂扫描版 PDF。

最推荐你用：

一份结构清晰的 PDF
或一份纯文本 / Markdown 文档
页数不要太多
内容主题尽量集中
这样做的好处是：

容易验证解析结果
chunk 质量更可控
更适合今天完成完整闭环
推荐学习和实战顺序
## 模块 1：先把 ingestion 流程拆清楚（40min）
这一阶段你先回答 4 个问题：

上传入口放在哪里
文档解析用什么方案
chunk 规则是什么
metadata 最少保留哪些字段
今天建议你的最小 metadata 至少包含：

source
page
chunkId
如果可以，再加：

fileName
chunkIndex
模块 2：实现上传和解析（1.5h）
这一阶段的重点是：

接住上传文件
成功读取文件内容
尽量拿到按页或按段的文本
你今天不需要支持所有格式，只要先支持一类稳定格式就够了。

目标不是“格式大全”，而是：

“至少有一种真实文档能进入系统。”

模块 3：实现 chunk 切分（1.5h）
当解析文本可用后，再做 chunking。

你今天应该先用简单、稳定的规则：

固定长度
可选 overlap
保持 chunk 不要过大
同时保留页码和顺序信息
这一步最关键的是：

chunk 不要太碎
chunk 不要大到塞进太多无关信息
每个 chunk 都要能追踪回来源
模块 4：完成 embedding 和入库（1.5h）
这一步把解析结果真正写进 Chroma。

你需要做的事：

为每个 chunk 生成稳定 id
准备 documents
准备 metadatas
执行 upsert
这里最关键的是：

同一个文档重复入库时，id 最好可复现
metadata 必须能支持后续引用显示
出错时要能知道失败在哪一段
模块 5：做一次完整入库验收（1h）
今天一定要留时间做完整验收，不要只写完代码就结束。

你至少要验证：

文档已被成功解析
chunk 数量合理
collection 条数有增加
peek() 能看到对应 source/page/chunkId
你知道这份文档到底入库了多少条
今天最值得注意的常见坑
坑 1：一上来就追求复杂 chunk 策略
Day 18 的目标是完整跑通，不是先把 chunk 策略调到最优。

坑 2：metadata 设计过少
如果你今天不保留 source/page/chunkId，后面做引用展示会非常被动。

坑 3：上传成功不等于入库成功
上传接口返回 200，只说明文件到了服务端，不代表文本解析和向量写入已经全部成功。

坑 4：没有记录中间产物
今天很值得把这些中间信息打印或保存出来：

解析出的总字符数
chunk 总数
前 3 个 chunk 示例
最终入库条数
