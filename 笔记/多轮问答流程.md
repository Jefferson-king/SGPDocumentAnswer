用户输入问题并点击发送按钮后，前端会触发 `AskPage.vue` 里的 `handleSubmit`。它会先校验当前问题是否为空、当前是否已有可用的已完成文档；如果可以发送，就从当前 `messages` 中整理出本次请求要携带的历史消息，只保留有效的 `user / assistant` 文本内容，并限制最近几轮，避免历史无限增长和上下文污染。与此同时，前端还会读取当前选中的 `documentId`，因为在今天新增的路由能力里，系统不仅要知道“用户问了什么”，还要知道“这次问题是否绑定某一篇具体文档”。随后前端会先创建一条用户消息和一条 assistant 占位消息，把它们插入 `messages`，这样页面会立刻显示“本轮提问”和“助手待回复”的消息框。接着调用 `askQuestionStream`，把 `question + history + documentId` 一起发给后端问答接口。

在前端的 `frontend/src/api/qa.js` 里，`askQuestionStream` 会通过 `fetch` 向 `/api/qa/ask` 发起请求，并使用 `response.body.getReader()` 持续读取后端返回的 SSE 流式数据，再用 `TextDecoder` 把二进制内容解码成文本。前端会按 `\n\n` 拆分出完整 SSE 事件，从每个事件中解析出 `data:` 后面的 JSON payload，再根据 `payload.type` 调用不同回调：如果是 `start`，除了把当前 assistant 消息的状态从 `thinking` 改成 `streaming` 之外，还会把本轮路由得到的 `mode` 和 `reason` 挂到当前消息上，这样前端就能直接展示“本轮采用：直接回答 / 检索回答 / 全文总结”；如果是 `citations`，就把这一轮回答对应的引用数据挂到当前 assistant 消息上；如果是 `delta`，就把新返回的一小段文本追加到 assistant 消息的 `content` 后面，因此答案会以流式方式一段一段显示出来；如果是 `done`，说明本轮回答结束，assistant 消息状态最终改为 `done`。

后端在 `src/routes/qa.js` 中接收这个请求。它会先从请求体里读取 `question`、`history` 和 `documentId`，校验问题不能为空，然后设置 SSE 响应头。和之前不同的是，现在路由层不再一开始就默认本轮一定走 RAG，而是先调用 `answerService.getAnswerStreamPayload({ question, history, documentId })`，把“本轮该走哪条链路”以及“具体如何执行”交给 service 层统一处理。等 `answerService` 返回后，路由层会先通过一个 `start` 事件把本次路由结果里的 `mode` 和 `reason` 返回给前端，再发送 `citations`，然后把模型流式生成的文本一段一段包装成 `delta` 事件持续写回，最后发送 `done` 事件并结束响应。这样一来，接口返回的不再只是答案本身，还显式暴露了“系统为什么这么回答”。

在 `questionRouterService.js` 里，系统会先根据问题内容做一次简单规则路由，这也是今天新增的核心能力。它会根据用户问题中的关键词和当前是否传入 `documentId`，判断这次请求属于哪一种模式：如果问题更像通用闲聊或常识提问，就走 `direct`；如果问题明显依赖知识库片段，例如“这份文档里有没有提到……”“原文怎么说”“哪一页提到……”，就走 `retrieval`；如果问题明确要求概括整篇文档、提炼重点、总结核心结论，就走 `summarize`。这个模块只负责输出 `mode` 和 `reason`，不直接生成答案，因此它本质上是一个独立的决策层，而不是回答层。这样做的好处是，问答逻辑和决策逻辑被拆开了，后续如果你想把规则路由升级成模型路由，也有明确的扩展位置。

在 `chatHistoryService.js` 里，后端会再次对前端传来的历史做规范化和裁剪，保证只保留合法的 `user / assistant` 消息，并限制最近几轮，避免前端绕过限制后把过长历史直接送进模型。除此之外，这一层还会做两件事：一是把最近几轮对话整理成给 prompt 使用的“历史对话块”，帮助模型理解“它”“第二点”“刚才那部分”这类追问；二是构造给检索使用的 `retrievalQuery`，把当前问题和最近几轮对话拼起来，让向量检索在多轮追问场景下更容易命中真正相关的文档片段。也就是说，多轮能力本身并没有被今天的路由改掉，而是被保留下来，继续为 `retrieval` 和 `summarize` 提供上下文支持。

在 `answerService.js` 里，系统会先根据 `questionRouterService` 的判断结果决定本轮执行路径。如果路由结果是 `direct`，它不会去检索知识库，而是直接组装普通对话 prompt，调用 `ollama.generateCompletionStream(prompt)` 返回答案；如果路由结果是 `retrieval`，它会先基于裁剪后的历史构造检索查询，再调用 `retrievalService.retrieve(...)` 去知识库中检索相关片段，随后把这些片段整理成结构化 `citations`，包括 `source`、`page`、`chunkId`、`snippet` 等信息，再把“历史对话 + 当前检索结果 + 当前问题”一起组装成最终 prompt，要求模型优先依据检索结果回答；如果路由结果是 `summarize`，它则不会只取少量命中片段，而是会根据当前 `documentId` 把该文档对应的全部 chunks 从向量库中取出来，按顺序拼成总结上下文，再组装成“全文总结型 prompt”，让模型对整篇文档做结构化概括。最后，无论走哪条链路，`answerService` 都会把结果统一包装成 `{ mode, reason, citations, stream }` 交回给路由层，因此上层接口不需要关心底层到底是直接回答、检索回答还是全文总结。

在 `retrievalService.js` 里，后端会先把检索查询转成 embedding，再调用 Chroma 做向量检索，返回命中的 chunk 内容和 metadata。和之前相比，今天这层新增了一个重要能力：它可以根据 `documentId` 做限定检索，也就是“只在当前选中文档内查找相关片段”，而不是永远在整个知识库里全局搜索。由于在文档入库阶段已经为每个 chunk 保存了 `documentId`、`source`、`page`、`chunkId` 等信息，所以这里检索回来的结果天然带着引用所需的数据，这也是前端后面能展示页码、片段摘要和 chunk 标识的基础。同样地，全文总结模式之所以能成立，也不是因为上传流程被改了，而是因为文档入库时本来就已经把 chunks 和 `documentId` 存进了向量库，现在只是问答阶段新增了“按文档聚合读取”的能力。

整个链路串起来后，现在一次完整的问答已经不再是“所有问题都固定走 RAG”，而是变成了：前端裁剪历史、携带当前问题和 `documentId` 发起请求；后端再次裁剪历史，并先通过独立路由模块判断本轮任务属于 `direct`、`retrieval` 还是 `summarize`；执行层再根据不同模式调用对应能力；接口通过 SSE 把 `mode`、`reason`、`citations` 和流式文本持续返回给前端；前端则一边展示“本轮采用的处理模式”，一边把引用和 `delta` 文本实时追加到当前 assistant 消息中。这样一来，项目就从“单一路径的 RAG 问答”升级成了“先决策、再执行”的最小 Agent 化问答系统。
