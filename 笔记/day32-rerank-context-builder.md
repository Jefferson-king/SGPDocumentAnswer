# Day 32｜2026-04-15｜4h

- 提升最终上下文质量

## 适用范围

- 当前检索状态：已具备向量召回或混合召回基础

- 检索链路更像正式项目而不是 demo
- 每个阶段的结果都能被打印和解释

## 先用一句话理解

今天要做的，是把“召回结束就拼 prompt”改造成“召回候选 -> 重排 -> 最终上下文”的三段式流程。

## 为什么今天一定要补 rerank

召回阶段最重要的是“不要漏”，不是“已经最优”。

所以真实 RAG 项目里，常见做法都是：

1. 先宽松召回更多候选
2. 再重排
3. 再挑少量最合适的上下文喂给模型

如果你不加 rerank，最容易出现的情况就是：

- 候选太散
- 虽然有正确 chunk，但排位不稳定
- prompt 里混进了太多次优内容

一句话版：

“今天补 rerank，不是为了链路更复杂，而是为了让最终上下文更像精挑细选的结果。”

## 推荐学习和实战顺序

### 模块 1：先定义三段式数据结构（1h）

今天最先要做的是把链路上的三个阶段显式命名。

建议你统一成这三个对象：

- `recallCandidates`
- `rerankedCandidates`
- `finalContext`

每个阶段都尽量保持统一字段：

- `id`
- `content`
- `metadata`
- `scores`
- `debugInfo`

今天的重点不是字段多华丽，而是做到：

- 每一层职责明确
- 后面调试时知道问题出在哪一层

### 模块 2：实现第一版 rerank（2h）

今天的 rerank 不需要一上来就接模型重排。

最稳的第一版通常可以基于这些信号：

- query 词覆盖率
- 词重合度
- 标题/章节命中加分
- 页级命中加分
- BM25 / vector 融合分数
- chunk 长度惩罚或奖励

也就是说，你完全可以先做一个“规则型 rerank”：

- 命中 query 关键词越多，分越高
- chunk 越贴近问题主题，分越高
- metadata 更完整、位置更优，分越高

如果今天时间够，可以额外预留：

- `modelRerank()` 接口

但今天先别被“模型重排”绑住，先把 rerank 这层的职责做出来更重要。

### 模块 3：做 final context builder（1h）

今天第三步是真正决定“最终喂给模型什么”。

你至少要控制：

- 最终保留几条
- 每条截多长
- 是否允许同页重复
- 是否允许相邻 chunk 全塞进去

这一步的目标不是简单 topN，而是尽量避免：

- 同一个页面重复 3 次
- 相邻 chunk 内容高度重叠
- 次优内容把 prompt 撑满

最稳的第一版策略通常是：

- 从 rerankedCandidates 依次选
- 遇到重复页/重复 chunk 相似内容就跳过
- 直到达到 `finalContextCount`

## 今天最值得注意的常见坑

### 坑 1：召回和最终上下文还是一个概念

这样 rerank 这一层就形同虚设。

### 坑 2：final context 只按分数截断

这样容易把重复内容塞满 prompt。

### 坑 3：没有打印三段式结果

后面你根本不知道问题出在 recall 还是 rerank。

### 坑 4：今天就急着做复杂模型重排

先把规则型 rerank 做稳，再升级更值。
