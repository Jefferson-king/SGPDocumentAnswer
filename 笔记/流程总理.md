```
1. 用户拖文件进区域
   ↓
2. 子组件触发 dragenter → 告诉父组件
   ↓
3. 父组件 dragActive = true → 传回子组件变蓝
   ↓
4. 用户松开鼠标（真正拿到文件）
   ↓
5. 子组件 onDrop → 获取文件
   ↓
6. 子组件把文件抛给父页面
   ↓
7. 父页面校验格式、大小
   ↓
8. 合法 → 显示“开始上传”按钮

```
## 前端流式读取链路
- AskPage:读取输入框，点击发送handlesubmit,先插入一条user消息，再插入一条空的assitant消息，状态为thinking,调用askQuestionStream，在onDelta中将增量文本不断增加到assitentMessage.content,在onDone和onError中处理结束和错误状态，调用scrollToBottom，保持滚动到底部
- qa.js:前端流读取层，fetch(/api/qa/ask),通过response.body.getReader()拿到流，进行textdecoder,按照\n\n切分sseEvent,提取data字段，根据payloadType字段分发给onStart/onDelta/onDone/onError.
- MessageList：纯展示层。接收来自父组件Askpage的messages,区分user/assitant,判断status：thinking/streaming/error,渲染不同的UI。若为thinking，显示正在思考；若为streaming,显示打字光标|，若为error，显示错误信息。
- 后端：src/routes/qa.js:接收前端POST/api/qa/ask请求，设置sse响应头，调用answerService.getAnswerStreamPayload(question)，获取sources及stream，并将stream返回给前端，先发送type'start'，返回sources,然后发送type"delta"，返回增量文本，最后发送type"done"，告诉前端结束。
- answerService.js:RAG组织层，先检索文档，将检索到的内容同question，角色设定约束条件组成prompt，把来源sources整理出来，调用ollma.generateCompletionStream(prompt) 拿模型流.
- ollma.js:模型流：负责请求ollama的接口/api/generate,开stream:true，持续读取模型返回的chunk,每拿到一段chunk就yield出去，直到模型结束
- retrievalService.js:检索层，调用retrieve(question,topK),将question转为embedding,从向量库检索与question最相关的topK个片段，返回id content metadata score。

4.10 工作总结
在 AskPage.vue 里，把原来的“单轮发送问题”升级成了“携带有限历史的连续对话”。现在前端会保留整段消息列表，但发送请求前会先用 toRequestHistory 裁剪出最近几轮有效消息，再把 question + history 一起发给后端，这样追问时模型能接住“第二点”“它”“刚才那部分”这类上下文。
在 AskPage.vue 里，补上了“新会话”能力。现在页面初始化会通过 createWelcomeMessage() 生成欢迎消息，清空时也会回到这个初始状态；同时给消息对象加了 includeInHistory 字段，欢迎语只展示、不参与历史，避免无意义内容混进上下文。
在 frontend/src/utils/chatHistory.js 里，新增了前端历史裁剪逻辑。现在会从完整 messages 里筛出真正要发给后端的消息，只保留 user / assistant 的有效文本内容，并限制最近若干轮，避免历史无限增长。

在 src/routes/qa.js 里，把问答接口从“只接收当前问题”改成了“接收当前问题 + 历史消息”。现在后端从请求体里除了读取 question，还会读取 history，再把这两部分一起交给回答服务处理。
在 src/services/chatHistoryService.js 里，新增了后端历史处理逻辑。现在服务端会再次对前端传来的历史做规范化和裁剪，并提供历史格式化、检索查询拼接等方法，保证真正送进检索和 prompt 的历史是有限且干净的。
在 src/services/answerService.js 里，把原来的“当前问题直接检索并回答”升级成了“有限历史 + 当前问题 + 每轮重新检索”。现在后端会先基于当前问题和最近对话构造检索查询，再把裁剪后的历史和当前检索结果一起组装进 prompt，同时明确要求模型优先依据本轮检索结果回答，避免只顺着旧答案继续说。

今天这次改动的核心，不是单纯把历史消息带过去，而是把“前端保留完整会话、请求只带有限历史、后端再次裁剪并继续检索”这条多轮链路接通了。这样项目从单轮 RAG 问答，正式升级成了可以连续追问 3 到 5 轮的多轮问答版本。
