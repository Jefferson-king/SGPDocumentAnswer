什么是今天的最小闭环
今天的最小闭环只包含 4 件事：

准备一小批主题分组明显的文本
写入 Chroma collection
发起一条查询
看返回结果是否和查询主题一致
一句话版：

“先别急着问模型回答得好不好，先确认数据库本身能把相关片段找回来。”

今天为什么一定要先做这个
因为后面的 ingestion、retrieval API、RAG 评测，全都建立在一个前提上：

你的向量库已经能稳定存数据
也已经能稳定召回数据
如果这一步没跑通，后面你会一直分不清问题到底出在：

文档解析
chunk 切分
embedding
向量库写入
还是 query 检索
Day 17 的意义，就是把“向量存储”和“向量检索”这件事单独跑通。

先建立今天的最小心智模型
你今天先把 Chroma 理解成：

它不是聊天模型
它不是答案生成器
它是一个“存向量并按相似度检索”的数据库
最小流程可以理解成：

你准备一批文本 chunk
每条 chunk 被转成向量
向量和原文一起写入 collection
你输入一句查询
查询也会被转成向量
系统按距离或相似度找出最接近的 chunk
返回前 k 条结果
一句话版：

“不是数据库看懂了文本，而是 embedding 把文本映射到向量空间后，数据库根据距离找近邻。”

今天最值得使用的最小数据集
今天不要一上来就用真实 PDF，全都手工准备更稳。

最推荐的数据集结构：

10～30 条短文本
至少分成 2 个主题组
每条 1～3 句
主题边界尽量明显
例如两组：

电网 / 低压 / 三相不平衡
前端 / Vue / 路由
这样做的好处是：

检索结果是否正确，肉眼很容易判断
不会被复杂文档解析问题干扰
适合先理解 query 行为
推荐学习和实战顺序
模块 1：把 Chroma 跑起来并确认连通（40min）
你今天先确认 3 件事：

Chroma 服务已经能正常启动
项目里已经记录 CHROMA_URL
Node 脚本已经能 getOrCreateCollection
这一阶段的目标不是写数据，而是回答一句话：

“我的 Node 项目已经能连上 Chroma 了吗？”

## 模块 2：准备最小数据集并设计数据结构（1h）
这一阶段先不要碰真实文档。

你应该先准备：

id
document
metadata
建议结构：

id：稳定、可复现，例如 ${source}-p${page}-c${chunkIndex}
document：原始文本 chunk
metadata：{ source, page, chunkIndex, topic }
这一阶段最关键的是：

id 不要随机
metadata 先保留最小但有意义的信息
文本主题要分组明显
## 模块 3：完成 upsert，验证 count 和 peek（1h）
当数据准备好后，再做真正写库动作：

获取 collection
执行 upsert
用 count() 看条数
用 peek() 看前几条数据
这一阶段的重点不是 query，而是确认：

写入成功
条数正确
原始数据结构没错
如果这一阶段不稳，后面 query 结果就没有可信基础。

模块 4：开始做最小 query 验证（1.5h）
当数据已经写进去后，再开始查。

建议先测这类查询：

“三相不平衡会带来什么问题？”
“Vue Router 是做什么的？”
你需要观察 3 件事：

返回的 chunk 是否和主题一致
返回顺序是否合理
top-k 改变时结果是否明显变化
这一阶段最关键的目标是：

“我能肉眼判断，相关内容确实被排在前面了。”

模块 5：理解距离、相似度、top-k（40min）
今天不需要数学推导，但你要建立基本直觉。

你至少要理解：

top-k：返回前多少条结果
distance：越小通常表示越接近
相似度：越高通常表示越相关
检索结果不是“绝对正确”，而是“相对接近”
这一步的目标是：

“我知道为什么 query 返回的是这些 chunk，而不是另一些。”

今天最值得注意的常见坑
坑 1：还没装 embedding 依赖就开始 upsert
如果你只传 documents，JS 客户端需要可用的 embedding function。否则会出现默认 embedding 提示或相关报错。

坑 2：id 每次都随机生成
如果 id 不稳定，你后面重复运行脚本时就会不断插入新数据，难以区分是更新还是重复写入。

坑 3：最小数据集主题不够分明
如果两组文本太像，你很难判断检索是好是坏。Day 17 应该先选边界清楚的数据。

坑 4：一开始就纠结召回“完美正确”
今天只要求建立最小检索闭环，不要求达到最终线上效果。

count() 能返回正确条数
