## 推荐学习和实战顺序

### 模块 1：统一响应结构（1.5h）

今天最适合先把问答返回结构固定下来。

推荐统一成：

- `answer`
- `citations`
- `summary`
- `confidence`

每个字段的职责也要写清楚：

- `answer`：主回答正文
- `citations`：引用数组
- `summary`：简要摘要或精简版结论
- `confidence`：低调使用的置信度或可信程度估计

今天重点不是“字段看起来高级”，而是让前后端以后都围绕同一协议工作。

### 模块 2：增加 schema 校验（1h）

今天最值得补的是输出校验。

当前项目最稳妥的方式通常是：

- 用 `JSON Schema` 做输出结构校验

如果以后拆 Python 服务，再考虑：

- `Pydantic`

今天第一版至少要做到：

- 缺字段时能发现
- 字段类型不对时能发现
- 前端不需要再到处猜某个字段是不是存在

### 模块 3：前端适配统一输出（1h）

当输出 schema 固定以后，前端也要一起收口。

建议展示职责明确分层：

- `answer` 主回答作为核心显示
- `citations` 放在回答下方
- `summary` 作为次级摘要显示
- `confidence` 低调展示，不喧宾夺主

今天的目标不是做更多 UI，而是减少前端“因为后端结构不稳而写的补丁逻辑”。

### 模块 4：录屏与总结（30min）

今天一定要把增强版链路跑一遍。

推荐演示顺序：

1. 上传一个 PDF/TXT
2. 等待入库 ready
3. 问一个事实题
4. 问一个总结题
5. 再追问一轮 followup
6. 展示引用、summary、confidence
7. 说明混合召回、rerank、结构化解析这些设计点

## 今天最值得注意的常见坑

### 坑 1：后端统一了结构，前端还继续兼容旧逻辑

这样项目会长期背双份债。

### 坑 2：confidence 写得过于绝对

RAG 系统里这个字段更适合低调提示，不适合夸大。

### 坑 3：schema 校验只写了文档，没真正执行

那它对稳定性帮助有限。

### 坑 4：今天只录成功演示，不准备讲解话术

可演示项目不只要能跑，还要能解释为什么这样设计。

## 本周结束后，你的项目应该具备的简历表达

- 将单一路径向量检索升级为“向量检索 + BM25”混合召回，提升关键词与语义联合命中能力
- 设计 `topK recall -> rerank -> final context` 三段式检索流程，提升答案准确率与稳定性
- 引入 chunk + 父块/页级召回机制，缓解长文档上下文碎片化问题
- 将文档处理从纯文本提取升级为结构化解析，支持标题、段落、表格、页码识别及 richer metadata
- 设计问答控制层，支持事实、总结、对比、追问等多类问题模板
- 统一模型输出为 `answer / citations / summary / confidence`，增强前后端协作效率与工程可维护性
