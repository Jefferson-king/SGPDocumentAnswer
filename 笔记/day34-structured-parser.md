# Day 34｜2026-04-17｜4h

## 今日主题

- 把纯文本提取升级为结构化解析
- 至少区分标题、段落、表格、页码
- 让 chunk 切分从字符长度导向转向 block 导向

验收：

解析结果已经不再只是“一整页文本”，而是结构化 block 列表。

## 适用范围

- 当前项目技术栈：`Node.js + Express + Vue`
- 当前解析状态：PDF/TXT 解析后多为页文本或长字符串
- 当前目标：为 richer metadata、parent recall、structured output 打下基础

## 今天做完后，你应该达到的结果

- 文档处理更像正式 RAG 项目
- chunk 不再只是按固定字符硬切
- 后续接 Docling 或更强 parser 时已有标准接口

## 先用一句话理解

今天要做的，是让“文档解析结果”从一段纯文本，升级为一组有结构、可追踪、可切分的 blocks。

## 为什么今天一定要做结构化解析

如果解析结果永远只是整页文本，你后面很多增强都会受限：

- 标题命中无法加分
- 表格无法单独处理
- 父块召回边界不清楚
- chunk 很容易把不同结构混在一起

一句话版：

“今天做结构化解析，不是为了炫技，而是为了让后面每一层增强都建立在更好的输入上。”

## 推荐学习和实战顺序

### 模块 1：先定义 block schema（1h）

今天最重要的第一步是统一 block 结构。

每个 block 至少建议包含：

- `id`
- `type`
- `text`
- `page`
- `section`
- `order`
- `parentId`
- `bbox` 或预留字段

其中 `type` 至少先支持：

- `title`
- `paragraph`
- `table`
- `pageNumber`

今天的重点不是识别率百分之百，而是让结果先具备“结构化表达能力”。

### 模块 2：做第一版规则解析（2h）

今天最稳的结构化解析，不是直接追求复杂 OCR/版面理解，而是先做“可用的规则版”：

- 标题：短句、编号模式、明显分节文本
- 段落：常规正文文本
- 表格：先做弱识别或占位块
- 页码：单独抽取

你完全可以接受今天的第一版不是最强，只要做到：

- 输出 block 列表
- block 类型有区分
- 结构在后续链路里可用

今天重点是“先把 parser 输出格式升级”，不是一次性把版面理解做到完美。

### 模块 3：重写 chunker 为 block-based（1h）

当 block 有了以后，chunker 的思路就应该升级。

过去是：

- 按字符长度切

今天开始更适合：

- 以 block 为基本单位切
- 尽量不把标题和正文混切
- 尽量不把表格和正文强行拼在同一个 chunk

最稳的第一版策略通常是：

- 优先按 block 聚合
- 再控制 chunk 上限
- 必要时才在 block 内二次切分

## 今天最值得注意的常见坑

### 坑 1：结构化解析结果还是只给 text，不给 type

那和没结构化差别不大。

### 坑 2：今天就想把表格识别做到很强

先做可用 schema，比强行做复杂识别更值。

### 坑 3：block 有了，chunker 还沿用原来的固定字符切法

这样前面的 parser 升级价值会被吃掉。

### 坑 4：没有保存解析结果样例

后面你会很难判断解析是不是越来越好。

## 今天的验收标准

- block schema 已经定义清楚
- 解析结果已经能输出 block 列表
- 至少能区分标题、段落、表格、页码
- chunker 已经开始基于 block 切分

## 建议的测试方式

- 拿一个标题明显的 PDF 测解析
- 看标题和正文是否已被分开
- 看表格是否至少被识别成独立 block 或占位 block
- 看 chunk 是否比之前更自然
