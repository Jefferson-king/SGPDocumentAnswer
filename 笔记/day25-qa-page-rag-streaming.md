## 适用范围

- 当前项目技术栈：`Node.js + Express + Vue`
- 当前数据状态：已有真实文档可上传并进入知识库
- 当前后端能力：已有 retrieval / RAG 回答接口基础
- 当前前端状态：已有上传页和页面骨架
- 当前目标：把“后端能回答”升级成“前端用户可以顺畅提问并实时看到回答”

## 今天做完后，你应该达到的结果

- 能完成一个正式的问答页
- 能接入后端 RAG 接口
- 能在前端实时显示流式回答
- 能展示多条消息而不是一次性输出
- 能让用户感受到“这是一个真正可用的问答产品页面”

## 先用一句话理解

今天要做的，不是再提升回答质量，而是把“用户提问 -> 检索 -> 生成 -> 流式显示”变成真正可交互的问答体验。

## 什么叫“核心问答体验可用”

一个可用的问答页，至少要满足：

- 用户知道从哪里输入问题
- 用户发送后能立即感知系统开始处理
- 用户能看到消息列表不断累积
- 用户能看到回答是逐步出现的
- 用户能区分自己的问题和系统的回答

一句话版：

“今天要让 RAG 不只是后端能力，而是用户真正能用起来的页面能力。”

## 今天为什么一定要做问答页

因为从今天开始，产品的核心价值第一次直接被用户感知：

- 上传页解决的是“把文档放进来”
- 问答页解决的是“知识库能不能真正帮我回答问题”

如果 Day 25 不完成，前面的入库和检索都还只是后台能力。

## 先建立今天的最小问答链路

你今天最适合把问答页理解成这条链路：

1. 用户输入问题
2. 前端把问题加入消息列表
3. 前端调用后端问答接口
4. 后端做检索和生成
5. 回答通过流式方式返回
6. 前端逐段追加内容
7. 回答结束后保留本轮消息

一句话版：

“问答页的本质，是把后端的 RAG 流程变成用户可以连续使用的消息体验。”

## 推荐学习和实战顺序

### 模块 1：先定页面最小组成（40min）

问答页最少要有 4 块：

- 标题或当前知识库提示
- 消息列表
- 输入框
- 发送状态区

今天先别急着加太多额外按钮，先保证提问主链路清楚。

### 模块 2：完成消息列表和输入区（1.5h）

你今天最该先做的是消息结构，而不是打字机效果。

推荐最小消息结构：

- `id`
- `role`
- `content`
- `status`

至少支持两类消息：

- user
- assistant

发送时建议先插入：

- 用户消息
- 一个空的 assistant 占位消息

这样流式内容才能持续追加到同一条消息上。

### 模块 3：接入 RAG 接口和流式读取（2h）

当页面结构稳定以后，再接流。

今天最推荐的思路是：

1. 前端用 `fetch`
2. 后端返回 SSE 或可读流
3. 前端用 `ReadableStream` 持续读取
4. 将增量内容追加到最后一条 assistant 消息

这里最关键的是：

- 不要等全部回答结束再一次性渲染
- 必须让用户立刻看到页面在“持续生成”

### 模块 4：补打字机效果和过程状态（1.5h）

当基础流式链路通了以后，再补体验层。

你今天最适合补的体验有：

- 正在思考提示
- 打字机式增量显示
- 发送按钮禁用状态
- 自动滚动到底部

这一步的目标是让问答页看起来像成品，而不是只会输出文本。

### 模块 5：补失败场景与空状态（40min）

今天至少要考虑：

- 没有上传文档时如何提示
- 问题为空时如何提示
- 请求失败时如何提示
- 流中断时如何收尾

这一步你先不追求完美，但一定要让页面不会“卡死无反馈”。

## 今天最值得注意的常见坑

### 坑 1：先做打字机效果，再做消息结构

这样后面非常容易返工。

### 坑 2：流式返回只追加纯文本，不维护消息状态

后面做引用和多轮时会很难扩展。

### 坑 3：没有 loading 状态

用户发送后如果页面几秒都没变化，会以为系统坏了。

### 坑 4：消息列表没有区分 user / assistant

问答页会立刻失去可读性。

## 今天的验收标准

- 页面能输入问题并发送
- 页面能显示 user 和 assistant 两类消息
- 回答能以流式方式逐步出现
- 页面不会因为慢响应而显得无反应
- 核心问答体验已经可用

## 建议的测试问题

- 文档里明确提到过的问题
- 文档中需要检索 2 个以上 chunk 的问题
- 文档里没有答案的问题

今天最关键的不是问得多，而是验证问答页的交互链路是否稳。
