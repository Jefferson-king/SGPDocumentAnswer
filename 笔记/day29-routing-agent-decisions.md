# Day 29｜2026-04-23｜5h

## 今日主题

- 增加简单路由能力
- 区分“直接回答”“检索文档后回答”“总结整篇文档”
- 把路由逻辑写清楚

验收：

项目不只是 RAG，还体现 agent 决策。

## 适用范围

- 当前项目技术栈：`Node.js + Express + Vue`
- 当前产品状态：Beta 版已经具备上传、问答、引用、多轮主链路
- 当前后端能力：已有普通聊天、RAG 检索回答、文档列表和上传能力
- 当前目标：让系统不再只有单一路径，而是能根据用户问题选择更合适的处理方式

## 今天做完后，你应该达到的结果

- 能解释什么叫简单路由能力
- 能区分至少 3 种处理模式
- 能把路由逻辑单独写清楚，而不是散落在问答接口里
- 能让项目开始体现 agent 决策感
- 能给后续接 LangChain 或更复杂 agent 奠定基础

## 先用一句话理解

今天要做的，是把“所有问题都走同一个 RAG 接口”升级成“系统会先判断这类问题该怎么处理，再走对应链路”。

## 什么叫“简单路由能力”

你今天先不要把路由想成复杂 planner。

对你当前项目来说，最小可用的路由能力只需要解决这个问题：

- 这个问题适合直接回答
- 这个问题适合先检索文档再回答
- 这个问题适合总结整篇文档

一句话版：

“Day 29 的核心，不是做一个大而全的 Agent，而是让系统开始具备最基本的决策分流能力。”

## 今天为什么一定要做这一步

如果所有请求都硬走同一个 RAG 流程，会很容易出现：

- 明明是通用问题，却也去检索文档
- 明明用户想总结整篇文档，却只做普通问答
- 系统看起来像一个单功能问答器，而不是有策略的 Agent

Day 29 的意义，就是让你的项目开始从“RAG 产品”向“有决策能力的 Agent 产品”迈一步。

## 先建立今天的最小决策链路

你今天最适合把路由理解成这条链路：

1. 用户提问
2. 系统先判断任务类型
3. 根据任务类型选择处理模式
4. 调用对应能力
5. 返回结果和模式信息

最推荐的 3 类模式：

- direct：直接回答
- retrieval：检索文档后回答
- summarize：总结整篇文档

一句话版：

“先决定走哪条路，再决定怎么生成答案。”

## 推荐学习和实战顺序

### 模块 1：先定义 3 类任务边界（50min）

今天最先要做的不是写分类器，而是先把边界写清楚。

你可以这样理解：

- direct：通用问题、不依赖知识库、或者用户只是在闲聊
- retrieval：需要依据知识库片段回答的问题
- summarize：用户明确要求概括整篇文档、全文总结、提炼重点

这一阶段你一定要先写出判断规则，否则后面很容易混乱。

### 模块 2：先做规则路由，不急着做模型路由（1.5h）

Day 29 最稳的做法不是一开始就让模型自动判断一切，而是先做简单规则路由。

例如：

- 包含“总结全文”“概括这份文档”“提炼重点”时走 `summarize`
- 明确提到“文档里有没有”“根据资料回答”“这份文档提到”时走 `retrieval`
- 其他基础问题默认走 `direct`

这样做的好处是：

- 容易调试
- 行为可解释
- 后续可以逐步升级成模型判断

### 模块 3：把路由逻辑从问答逻辑中拆出来（1h）

今天最值得做的工程动作是：

- 单独写一个 router service 或 decision module

例如：

- `routeQuestion(question, options)`

它只负责输出：

- `mode`
- `reason`

后面的执行层再根据模式选择：

- 直接聊天
- RAG 问答
- 文档总结

### 模块 4：给前端或日志展示当前模式（1h）

如果路由已经做了，但用户和你自己都看不到系统到底选了哪条路，调试会很痛苦。

所以今天最值得补的是：

- 接口返回本次 `mode`
- 或前端展示“当前采用：检索回答 / 全文总结 / 直接回答”

这样别人才能看见你的系统确实在做决策，而不是只有一条黑盒链路。

### 模块 5：做一轮对比测试（40min）

今天至少要测这 3 类问题：

- “今天上海几点？” -> direct
- “这份文档里有没有提到无功补偿？” -> retrieval
- “帮我总结这份文档的核心内容。” -> summarize

这一轮测试的重点不是最终文案多漂亮，而是：

- 模式有没有选对
- 执行链路有没有走对
- 接口有没有把模式暴露出来

## 今天最值得注意的常见坑

### 坑 1：一上来就做复杂智能路由

Day 29 更稳的路径是先做规则路由，再考虑模型路由。

### 坑 2：路由逻辑写死在 controller 里

后面一扩展就会越来越乱。

### 坑 3：没有暴露当前路由模式

这样产品虽然在分流，但别人看不出来，也不好调试。

### 坑 4：所有非 direct 问题都强行走 retrieval

这会让“全文总结”这种任务体验很差。

## 今天的验收标准

- 系统至少支持 3 种处理模式
- 路由逻辑独立清楚
- 接口或前端能看到本次选中的模式
- 项目不再只是单一路径 RAG
- 产品开始体现 agent 决策感

## 建议的测试问题

- “现在上海几点？” 
- “这份文档里有没有提到 React Server Components？”
- “帮我概括这份文档的核心结论。”

这些问题最适合验证 3 种路由是否都能命中。
