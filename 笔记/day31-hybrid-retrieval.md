# Day 31｜2026-04-14｜4h

## 今日主题

- 把单一路径向量检索升级为向量检索 + BM25 混合召回
- 输出统一候选集合
- 能对比“仅向量”和“混合召回”的差异

验收：

检索链路已经支持 `vector + BM25` 双路召回，并能把结果合并成统一候选格式。

## 适用范围

- 当前项目技术栈：`Node.js + Express + Vue`
- 当前检索状态：主要依赖 embedding + Chroma 向量检索
- 当前目标：补上关键词精确命中能力，让术语、缩写、编号、规则词检索更稳

## 今天做完后，你应该达到的结果

- 召回不再只依赖 embedding
- 对精确词、术语词、缩写词问题更友好
- 检索结果能统一成一套候选格式，方便后面 rerank

## 先用一句话理解

今天要做的，不是直接追求最终答案更强，而是先让召回阶段不再只有一种路径。

## 为什么今天要补 BM25

向量检索很强，但它天然更偏语义相似。

真实文档问答里，最容易被纯向量检索吃亏的情况是：

- 专有名词
- 型号编号
- 简称缩写
- 规则词精确匹配
- 用户问法和文档原词高度一致

这类问题如果只有向量检索，召回经常不是不相关，而是“不够稳”。

一句话版：

“今天补 BM25，不是推翻向量检索，而是让召回阶段同时具备语义命中和关键词命中。”

## 推荐学习和实战顺序

### 模块 1：先接入 BM25（1.5h）

今天第一步是让项目具备一个基础 BM25 通道。

你最适合做的第一版不是追求最复杂，而是先实现：

- 基于现有 chunk 文本建立 BM25 索引
- 支持 query 返回 topK 候选
- 返回和 vector retriever 接近的数据结构

最稳的思路通常是：

- 入库阶段把 chunk 文本和 metadata 保留下来
- BM25 索引至少能关联 `id -> content -> metadata`
- query 时返回 `id / content / metadata / bm25Score`

今天先做“能跑通”比“算法多高级”重要。

### 模块 2：做双路召回合并（1.5h）

当 BM25 和向量检索都能跑以后，下一步是把结果合并。

你今天最应该统一的数据格式至少包括：

- `id`
- `content`
- `metadata`
- `vectorScore`
- `bm25Score`
- `source`
- `page`

去重原则也要先定清楚：

- 同一个 `chunkId` 只保留一条
- 如果同一 chunk 同时被 vector 和 BM25 命中，就把两边分数都保留

一句话版：

“今天不是简单拼两个数组，而是把双路召回变成统一候选集合。”

### 模块 3：做第一版融合策略（1h）

今天最稳的融合策略不一定是最复杂的。

你可以先从这两种里选一个：

1. `rank fusion`
2. `归一化后加权融合`

如果你时间紧，最推荐先做 `rank fusion`，因为：

- 对分数尺度不敏感
- 实现简单
- 调试成本低

如果你时间够，再升级为：

- vector 权重 0.6
- BM25 权重 0.4
- 或对不同 query 类型动态加权

今天先做到“结果可解释、能打印、能比较”最重要。

## 今天最值得注意的常见坑

### 坑 1：BM25 结果和向量结果格式不一致

后面 rerank 会很难接。

### 坑 2：只做召回，不做去重

最后 prompt 里会出现重复 chunk。

### 坑 3：把分数直接硬加

vector score 和 BM25 score 常常不在一个尺度上。

### 坑 4：今天做完以后没有对比实验

没有“仅向量 vs 混合召回”的对比，今天的优化就很难证明。
