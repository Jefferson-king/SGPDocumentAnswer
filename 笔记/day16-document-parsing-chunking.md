## 什么叫“结构化 chunk”

今天的 chunk，不应该只是一个字符串数组。

对你当前项目来说，一个合格的结构化 chunk，最少应该包含这些信息：

- `id`
- `text`
- `source`
- `page`
- `chunkIndex`
- `startOffset`
- `endOffset`

如果你愿意再做稳一点，还可以补：

- `documentId`
- `documentName`
- `fileType`
- `tokensEstimate` 或 `length`

一句话版：

“chunk 不只是切下来的文本，还要带着它从哪里来、在文档哪里、它是第几个。”

### 模块 1：先选文档解析方案，不急着做复杂抽象（50min）

## 你今天真正要串起来的 3 个模块

### 1. 文档解析

文档解析负责做两件事：

- 识别输入类型
- 提取尽可能完整的原始文本

在你当前项目里，这一层通常对应：

- 文件读取逻辑
- `TXT` 解析器
- `PDF` 解析器

这一层你要确认的是：

- 不同文件类型能否走统一入口
- 提取结果是否统一成同一数据结构
- 失败时是否能明确报错

### 2. 文本清洗

文本清洗负责把“能读出来”变成“更适合切块”。

这一层通常会处理：

- 多余空行
- 连续空白
- 页眉页脚噪声
- 断裂得太奇怪的换行

这一层你要确认的是：

- 不要过度清洗，避免破坏原文语义
- 清洗规则要稳定、简单、可解释

### 3. Chunking

chunking 负责把长文本切成后续检索和引用都更友好的片段。

这一层你要确认的是：

- 长度规则是否清楚
- overlap 是否合理
- chunk 是否尽量语义完整
- metadata 是否足够支撑后续引用来源

## 今天最推荐的接口或函数策略

为了尽快拿到 Day 16 的结果，最推荐的不是一开始就做很散的脚本，而是尽量收敛成 1 条主处理链路。

推荐思路：

- 保留一个底层函数：`parseDocument(filePath)`
- 保留一个分块函数：`chunkDocument(parsedDocument, options)`
- 再封装一个主演示入口：`ingestDocument(filePath)`

这个主演示入口只做一件事：

- 把“读取文档 + 提取文本 + 分块输出”完整跑完

这样做的好处是：

- 调试路径清晰
- 后面接 embedding 更顺手
- 逻辑上也更符合正式 RAG 管道

## 最小后端心智模型

你今天最适合把服务端或脚本逻辑理解成下面这段伪代码：

```js
async function ingestDocument(filePath) {
  const parsed = await parseDocument(filePath);
  const cleanedPages = cleanExtractedText(parsed.pages);
  const chunks = chunkDocument(cleanedPages, {
    chunkSize: 700,
    overlap: 100,
  });

  return chunks.map((chunk, index) => ({
    id: `${parsed.documentName}-${index}`,
    text: chunk.text,
    source: parsed.documentName,
    page: chunk.page,
    chunkIndex: index,
    startOffset: chunk.startOffset,
    endOffset: chunk.endOffset,
  }));
}
```

你今天不一定要和这段伪代码 100% 一样，但脑子里最好有这个图景：

- 先解析
- 再清洗
- 再切块
- 最后统一输出结构化结果

## Chunk 规则最小建议

如果你今天不想在参数上纠结太久，可以先用一版够稳的默认规则：

- `chunkSize`：`500~800` 字符
- `overlap`：`80~120` 字符
- `TXT`：按全文切块
- `PDF`：优先保留页级边界，再在页内继续切块
- `source`：记录文件名
- `page`：`TXT` 可设为 `null`，`PDF` 记录真实页码

你先不要把规则想得太复杂。

今天的核心不是“找到理论最优值”，而是先拿到一套可以解释、可以演示、可以继续往后接的规则。

## 一个推荐的 chunk 结构示例

```json
{
  "id": "demo.pdf-p3-c1",
  "text": "这里是一段被切分后的正文内容……",
  "source": "demo.pdf",
  "fileType": "pdf",
  "page": 3,
  "chunkIndex": 1,
  "startOffset": 520,
  "endOffset": 1180,
  "length": 660
}
```

这个结构的价值在于：

- 后面做 embedding 很方便
- 后面做检索引用很方便
- 后面前端展示“答案来自哪一页”也很方便

## 今天最值得修的常见坑

### 坑 1：一上来就追求支持很多格式

如果今天你同时想搞：

- `TXT`
- `PDF`
- `DOCX`
- `HTML`
- 网页抓取

那大概率会把今天做散。

Day 16 最好先稳住 `TXT + PDF`。

### 坑 2：文档解析和分块耦合得太死

如果你把“读文件”和“切块逻辑”全写死在一起，

后面想换 `PDF` 解析库、换 chunk 策略、加清洗步骤时会很难改。

今天最好把它们拆成至少两层：

- 解析
- 分块

### 坑 3：只保留文本，不保留来源信息

如果 chunk 只有：

```json
{ "text": "..." }
```

那你后面虽然能做 embedding，但很难：

- 回答时引用来源
- 定位错误 chunk
- 做检索调试

Day 16 一定要开始建立 metadata 意识。

### 坑 4：chunk 太机械，切完读起来很碎

如果你只是按固定长度硬切，

很容易出现：

- 标题切一半
- 定义和解释分开
- 一句话切断到两个 chunk

今天即使先做简单版，也要有 overlap，并尽量让 chunk 保持可读性。

### 坑 5：PDF 提取正常，但页码丢了

如果你把整份 `PDF` 合并成一大段纯文本，

后面虽然还能切块，但你很难回答：

- 这一段来自哪一页
- 检索命中的内容该怎么回跳原文

对 `PDF` 来说，页级信息非常值得保留。

### 坑 6：只测成功路径，不测脏文档

你今天至少应该测 3 类情况：

- 普通 `TXT`，结构简单
- 普通 `PDF`，能正常提取
- 提取质量一般的 `PDF`，看看清洗和分块结果是否还能接受

## 今天的验收标准

- 能解释文档解析、文本清洗、chunking 三者如何衔接

为了提高成功率，今天不要一开始就拿特别复杂的扫描版文档。

- 一个内容干净的 `.txt`

这些材料的共同点是：

## 建议的观察重点

你跑完一次真实文档后，最值得盯着看的不是“有没有输出数组”，而是这些问题：

- chunk 数量是不是明显过多或过少
- 每个 chunk 的开头和结尾是否自然
- overlap 是否真的保留了上下文连续性
- `PDF` 页码有没有正确保留下来
- 同一页中相邻 chunk 是否能看出连续关系
